# Explorer

Wander a logical section of the **personas-web** codebase, surface exactly **10 items** worth fixing, let the user triage, then execute the accepted ones in-session. Designed for frequent / low-friction use — daily wandering — and pairs with `/research` (external sources) and `/architect` (heavy structural change).

This skill is **personas-web-specific.** It uses `.claude/CLAUDE.md` (project rules) and `.claude/design.md` (design system) as the authoritative reference, optionally `.claude/codebase-context.md` if `/context-map-generator` has been run, and the Obsidian vault under a `web/` namespace for run records, coverage tracking, and cross-run learning.

## Interaction conventions

Built for parallel CLI control — every user prompt is single-keystroke answerable.

- **Every prompt is a numbered menu.** Numeric input picks the option; **Enter** triggers the default; option `1. other → …` is the deviation lane (free text).
- **Every phase output (intermediate or final) ends with a `Next?` block** of 2–5 numbered next-step actions. Replying with a digit advances the run without typing prose.
- Long free-text answers are still accepted everywhere; the menu just makes the common case instant.

## Input

Ask **two** numbered-menu questions, in this order.

### Q1 — Area

```
Area? (Enter = pick for me)
  1. other → type a hint (path fragment, keyword, or app route)
  2. dashboard      (src/app/dashboard/* — demo dashboard, mocks)
  3. guide          (src/app/guide/* + src/data/guide/*)
  4. roadmap        (src/app/roadmap + Supabase voting)
  5. marketing      (features, how, compare, templates, connections)
  6. info           (blog, changelog, community, legal, security)
  7. playground     (src/app/playground/* — interactive demos)
  8. components     (src/components/* — shared UI primitives)
  9. shell          (app/layout, template, error, not-found, instrumentation, etc.)
  10. pick for me   ← default
```

Numeric options 2–9 each map to a coherent set of files. Option 1's free text falls through to the Phase 2a resolver (path fragment, keyword, or app-route segment). Option 10 / Enter triggers Phase 2b auto-pick.

### Q2 — Category

```
Category? (Enter = any)
  1. other → describe (free-form intent; layered onto an auto-picked category)
  2. any            ← default
  3. quality
  4. dx
  5. ui
  6. perf
  7. bug
  8. i18n
  9. a11y
  10. sec
```

Wait for both answers. Don't ask anything else upfront — further questions only if a phase requires clarification.

If the user replies just "go" or "wander" or types `/explorer` with no arguments, treat as "pick for me" + "any" (Enter defaults for both).

## Constants

- **Codebase reference files** (always loaded):
  - `.claude/CLAUDE.md` — project rules (i18n contract, semantic Tailwind, animation gating, React 19 purity, Sentry PII, Supabase guards, commit format).
  - `.claude/design.md` — design tokens, color, typography, spacing, motion.

- **Severities** — `critical | high | medium | low`
- **Effort buckets** — `xs (<15m) | s (15-60m) | m (1-3h) | l (>3h)`

## Phase 1: Load context & memory

### 1a. Required-file check

For each of `.claude/CLAUDE.md` and `.claude/design.md`:
- If missing → stop and instruct the user (CLAUDE.md is critical; design.md is required for UI categories).

### 1b. Read in order

1. `.claude/CLAUDE.md` — project rules (i18n, design tokens, animation gating, React 19 purity, Sentry PII, Supabase guards).
2. `.claude/design.md` — design system canonical reference.
3. `.claude/codebase-context.md` (if present) — area resolver.
4. `$VAULT/Architect/web/strong-patterns.md` (if present) — to know the canonical shapes the codebase has been observed to do well. When you propose a fix in Phase 5, **prefer the shape of an existing strong pattern** over inventing a new one. Reference the pattern in the item's `strong_pattern_ref` field.
5. `$VAULT/Patterns/explorer-web-preferences.md` — to deprioritize finding shapes the user has rejected before.
6. `$VAULT/Explorer/web/state.md` — to know what *other* explorers are working on right now.
7. `$VAULT/Explorer/web/coverage.md` — to know last-visit dates and yield per area.
8. `$VAULT/Explorer/web/passes.md` — to know which items were already rejected per area.
9. The 3 most recent files in `$VAULT/Lessons/` matching `*-explorer-web.md` (sorted descending) — to absorb recent self-reflection.

### 1c. Stale-claim sweep

In `$VAULT/Explorer/web/state.md`, any entry whose `claimed_at` is older than 2 hours is **stale** — assume the run was abandoned. Remove stale entries before proceeding. This keeps the file honest without an explicit lock.

### 1d. Snapshot freshness

If `.claude/codebase-context.md` is present, parse the `Generated:` line. If >30 days old OR `git rev-list --count HEAD` has advanced by >200 since the snapshot footer, warn but continue:
```
Warning: codebase-context.md may be stale ({N} commits / {D} days since last refresh).
Consider running /context-map-generator after this session.
```

If absent, proceed using filesystem area definitions (per Q1 menu).

## Phase 2: Pick area

### 2a. If user gave a hint (Q1 option 1 or specific number)

Numeric (2–9): the area is fixed; jump to 2c.

Free text (option 1): resolve the hint:
- App-route segment (e.g. `dashboard/agents`, `guide/[category]`) → matching files under `src/app/`.
- Path fragment (e.g. `lib/sentry-pii`, `data/guide`) → files matching the fragment.
- Component name (e.g. `Navbar`, `GlowCard`) → that file plus its consumers.
- Keyword (e.g. `framer-motion`, `useReducedMotion`, `mockApi`) → files with the keyword.

If the resolution is ambiguous (>3 plausible areas), present a short numbered list and ask "which one?" before continuing.

### 2b. If user said "pick for me" (option 10 / Enter)

Score each area by:
- **Staleness** — days since last visit per `coverage.md` (more days = higher score). Never-visited = max staleness.
- **Past yield density** — items accepted / items surfaced in last 1–2 visits (higher = higher score). Tie-breaker.
- **Active claim penalty** — if the area appears in `state.md` Active section, score = 0 (skip it; pick a different area).

Pick the top-scored area. If multiple tie, pick the one with the smaller file count (faster to scan, tighter feedback loop).

Tell the user which area you picked and why (one short sentence), then a `Next?` menu:

```
Next?
  1. other → name a different area or path hint
  2. proceed with {picked-area}   ← default
  3. abort
```

### 2c. Category filter

If the user's category filter is not `any`, narrow the scan focus accordingly. The area stays the same; the filter only changes what kind of items count toward the 10-item budget.

## Phase 4: Wander the code

Read enough of the area to identify 10 items. Budget your tool calls — don't read every file in a 100-file area. Sample strategically.

### 4a. Sampling strategy

For an area with N files:
- N ≤ 5: read all of them.
- 5 < N ≤ 20: read all entry-point files (`page.tsx`, `layout.tsx`, top-level component) + a random sampling of the rest, capped at 10 file reads.
- N > 20: read all entry points + grep-discover the largest files (`Glob` then sort by line count) + sample 5–8 of those.

Use `Read` with offset/limit when files are >500 lines — read top + bottom + a middle slice rather than the full file.

### 4b. What to look for, by category

For `quality`:
- Dead code, unreachable branches, unused exports.
- Duplicated logic across files (3+ near-identical blocks).
- Misleading names, unclear intent, leaking abstraction.
- Comments that explain "what" instead of "why" — flag the comment, not just the code.
- Commented-out code older than current branch.

For `dx`:
- Repeated try/catch boilerplate that should use a shared error helper.
- Missing error context (errors thrown without enough info to debug).
- Module boundaries violated (e.g. `data/` importing from `app/`).
- Type-unsafe API call sites (raw `fetch` without typed wrappers, when a typed wrapper exists).
- Build-time hot-paths (large bundles, slow rebuilds) — use `npm run build` output if recent.

For `ui`:
- Raw Tailwind classes where semantic tokens exist (per `.claude/design.md` and CLAUDE.md §Conventions). Surface high-density offenders.
- Visual bugs (overflow, alignment, contrast). Only flag if you can reproduce or strongly suspect from the code.
- Inconsistent spacing/radius/shadow vs the design tokens.
- Missing loading / empty / error states on user-facing components.
- Hydration mismatch hazards (client-only data leaking into server-rendered output).

For `perf`:
- Components running canvas/GPU/`requestAnimationFrame` without `useReducedMotion` (CLAUDE.md §3 — also a hard a11y violation; double-flag).
- Large lists without virtualization.
- Object/array literals in deps causing unnecessary re-renders.
- `useEffect` chains where one effect depends on another's state (cascade).
- Hydration boundaries that ship a too-large client tree (look for `"use client"` at high-up boundaries that should be lower).
- Unbounded particle/orb effects, heavy framer-motion variants on frequently re-rendering components.

For `bug`:
- React 19 purity violations: synchronous `setState` inside `useEffect` body, `Math.random` / `Date.now` / `new Date()` in render or `useMemo` factory (CLAUDE.md §4).
- Race conditions (state read-then-write without a transaction, async effects without abort).
- Edge cases unhandled (empty arrays, null/undefined, NaN).
- Stale closures in effects/callbacks.
- Off-by-one, boundary errors.
- Errors swallowed silently (catch with empty body or just `console.log`).
- Supabase access without the URL+anon-key guard (CLAUDE.md §6).

For `i18n`:
- Hardcoded English in JSX, `placeholder`, `title`, `aria-label`, `alt`, page `metadata` (CLAUDE.md §1).
- Locale files that are out of lockstep — keys present in `en.ts` but missing in any of the 13 non-en locales (or vice versa).
- English placeholders shipped in non-en locales (forbidden — every user-visible string ships translated).
- **Bundle nearby strings** — when a single component has 4+ hardcoded strings, surface ONE finding for "extract `<Component>` strings to en.ts + translate to all 14 locales" rather than 4 separate findings.

For `a11y`:
- Text utilities with opacity below `/60` (CLAUDE.md §2 + `custom-a11y/no-low-text-opacity` lint rule).
- Animation surfaces that don't honor `useReducedMotion` (CLAUDE.md §3 + `custom-animation/require-animation-gating`).
- Missing labels on form inputs.
- Keyboard navigation broken (clickable divs without role/tabIndex).
- Missing focus styles.
- Modal without focus trap, escape handler, or backdrop click.

For `sec`:
- Sentry PII leaks: `Sentry.captureException(err, { extra })` with un-scrubbed UUIDs, URLs, names, emails. Cross-check against `src/lib/sentry-pii.ts` SENSITIVE_FIELDS.
- Supabase `service_role` appearing anywhere in `src/` (CLAUDE.md §6 — forbidden).
- Externally-reachable API routes without auth/validation.
- User input directly interpolated into SQL/command strings.
- `.env*` files committed (other than `.env.example`).
- Auto-promote sec findings to severity `critical`.

### 4c. Honor the deprioritization signals

- If `Patterns/explorer-web-preferences.md` contains a rule like "user rejects cosmetic CSS findings without a measurable issue," skip those.
- If `Explorer/web/passes.md` for this area lists items by short fingerprint (file:line + 1-line summary), skip exact matches. A near-match is OK to surface — but note "previously passed; resurfacing because <reason>".

### 4d. Stop conditions

- 10 items found → stop scanning, move to Phase 5.
- Exhausted the area without 10 items → widen scope by pulling in the *adjacent* area (e.g. `dashboard` widens into `components` if dashboard-specific components live there). Note the widening in the run record. If still <10 after widening twice, stop with what you have and explain the shortfall.
- Tool budget exceeded (>40 file reads) → stop with what you have.

**Do not pad the list** with low-value items just to hit 10. Quality over quota.

## Phase 5: Categorize and structure each item

For each of the 10 (or fewer) items, capture:

```yaml
- id: 1
  title: "<short imperative phrase, ≤60 chars>"
  category: quality | dx | ui | perf | bug | i18n | a11y | sec
  severity: critical | high | medium | low
  effort: xs | s | m | l
  anchor: "<file_path>:<line_number>"
  evidence: "<2-3 sentence explanation of the gap, with verbatim code snippet if helpful>"
  suggested_fix: "<1-2 sentence shape of the fix — not the fix itself>"
  strong_pattern_ref: "<wikilink to Architect/web/strong-patterns#... entry>" | null
  i18n_impact: "<none | adds keys to en.ts (and 13 other locales) | touches existing keys (must update all 14 locales)>"
  cluster_hint: "<other ids that ship naturally with this one, or 'standalone'>"
```

**i18n_impact note:** in personas-web, *every* key change requires updating all 14 locales in the same commit (CLAUDE.md §1). Always quantify the locale work in the impact field.

### Severity rubric

- **critical** — security gap, PII leak, broken golden path, build failure on master.
- **high** — wrong behavior on the golden path, broken on a common edge case, regression risk if left.
- **medium** — paper cut, confusing UX, small perf hit, latent risk.
- **low** — polish, nice-to-have, taste-level.

If you find yourself rating most items "high," recalibrate downward. A 10-item list typically lands as 0–1 critical, 2–3 high, 4–6 medium, 1–3 low.

### Cluster detection

- Same file → same PR.
- Type/component dependency → ship in order.
- Same i18n component bundle → one extraction PR (and one full-locale-translation pass).

## Phase 7: Triage

Ask the user:
```
Which to action? Reply with item numbers (e.g. "1, 3, 4").

Shortcuts:
  all     — accept every surfaced item
  none    — accept nothing (still write the sweep note)
  ask     — guided walkthrough item-by-item
  Enter   — same as "none"   ← default
```

For each accepted item, execute it **in this same session**.

### Execution rules

**Single accepted item with a clear anchor (Option A):**
1. Apply the edit at `anchor`.
2. Run validation:
   - TypeScript → `npm run typecheck`
   - Frontend → `npm run lint` (warnings OK; errors must be fixed)
   - i18n key change → typecheck enforces the `Translations` interface shape; verify all 14 locale files compile.
   - Build sanity → `npm run build` (only if a routing or config change; expensive otherwise)
   - e2e → `npm run test:e2e` (only if a touched file is exercised by a Playwright spec)
3. **Stage only the paths this item touched** — `git add path/one path/two`. Never `git add -A`, `git add .`, or `git add -u`.
4. Commit atomically using the personas-web format (CLAUDE.md §7):

   ```
   explorer/<area>: <short title>

   Why: <root-cause or value>
   Risk: <low|med|high + rationale>
   Verified: <typecheck|lint|build|playwright>

   Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
   ```

**2+ accepted items (Option B):**
1. Print the inline plan (one paragraph per item: file, change shape, validation).
2. Execute in **risk-ascending order** (xs effort first, l last; severity ties broken by category — `bug` before `perf` before `i18n` before `quality`).
3. Atomic commit per item. Validation per commit.
4. If validation fails → fix inline, do NOT stack failing commits. No `--no-verify`, no `--amend`.
5. If a downstream item turns out to be redundant after an upstream commit, drop it and note the drop in the run record.

**Item that needs more thought (Option D — escape hatch):**
Record it in the run record as `decided: deferred` with the reason. Use sparingly.

### Frontend changes — non-negotiable (per CLAUDE.md)

If any accepted item touches `src/**/*.tsx`:
- **i18n contract:** all user-facing strings via `useTranslation()` + keys in `src/i18n/en.ts` + matching translations in all 13 non-en locales in the same commit. No hardcoded English in JSX, `placeholder`, `title`, `aria-label`, `alt`, `metadata`.
- **Semantic Tailwind:** `text-foreground`, `bg-background`, `bg-surface`, `text-muted-dark`, `border-glass`, `text-brand-cyan`, `rounded-*`. No `text-white`, `bg-black`, raw hex; no opacity modifiers below `/60`.
- **Animation gating:** any file using `requestAnimationFrame` / `cancelAnimationFrame` / canvas motion must import + call `useReducedMotion` and short-circuit.
- **React 19 purity:** never sync-`setState` in `useEffect`; never call impure functions in render or `useMemo`.
- **Sentry PII:** route new `captureException`/breadcrumb data through `src/lib/sentry-pii.ts`.
- **Supabase guards:** every Supabase call must check URL + anon key.

If you can't honor these in the change, defer the item — don't ship it half-converted.

### Visual verification

If a change is visually meaningful (UI category, or any change to a rendered component shape), state explicitly that you have NOT visually verified, OR start `npm run dev` and exercise the affected route in a browser before committing. Don't claim "looks good" from code review alone.

## Notes on use

- **Pair with `/research`** — run `/explorer` after a research session that touched a specific area, to immediately surface adjacent gaps.
- **Cadence** — daily or every-other-day is a reasonable rhythm. Coverage.md will tell you when the codebase is uniformly fresh and you should switch to `/architect` instead.
- **Coexist with uncommitted work.** Multiple CLIs and editor sessions share the working tree. Explorer never stashes, resets, or discards anything it didn't author. Each commit stages **only the specific paths** the explorer touched. Forbidden at all times: `git stash`, `git reset --hard`, `git restore`, `git checkout --` on paths the run didn't author, `git clean -f`, `git add -A` / `git add .` / `git add -u`.
- **Locale lockstep is real.** Any i18n key change ships with all 14 locales updated in the same commit. If you can't hand-translate (or you don't have a translation strategy), defer the i18n item.
- **Drift signal** — if 3+ explorer runs in a row produce 0 accepted items, the calibration is off. Trigger a self-reflection: read the last 3 sweeps and ask the user "what shape would have actually been useful?"
