# Architect

Heavy-hitter codebase scan for **structural patterns** in **personas-web** — both weak ones to upgrade and strong ones to codify. Designed for rare, deliberate, high-effort sessions where the payoff is a class of bugs eliminated, a tech swap landed, or a convention promoted from "tribal knowledge" to "lint-enforced rule."

This is the highest-risk, highest-payoff skill in the suite. It pairs with `/research` (external sources) and `/explorer` (per-area paper cuts) — those handle the small and the medium; `/architect` handles the large.

This skill is **personas-web-specific.** It uses `.claude/CLAUDE.md` and `.claude/design.md` as the authoritative reference, optionally `.claude/codebase-context.md` if `/context-map-generator` has been run, and the Obsidian vault under a `web/` namespace for a durable backlog of architectural decisions that span multiple sessions.

## Interaction conventions

Built for parallel CLI control — every user prompt is single-keystroke answerable.

- **Every prompt is a numbered menu.** Numeric input picks the option; **Enter** triggers the default; option `1. other → …` is the deviation lane (free text).
- **Every phase output (intermediate or final) ends with a `Next?` block** of 2–5 numbered next-step actions. Replying with a digit advances the run without typing prose.
- Multi-finding triages use `<id>=<verdict-number>` syntax (e.g. `1=2 2=1 3=3`); `all=<n>` and `ask` shortcuts are always accepted.
- Long free-text answers are still accepted everywhere; the menu just makes the common case instant.

- **Payoff** — 1 (incremental) … 5 (eliminates a recurring bug class or unblocks a major future)

## Phase 3: Parallel scan (scan + area modes)

Spawn **3–5 `Explore` sub-agents in parallel**, each looking at the theme/area from a different angle.

### 3a. Pick the angles

For a generic theme, default angles:
1. **Usage map** — where does this concept appear? Count call sites, group by app route / component module.
2. **Type/contract** — are the types consistent? Are interface boundaries respected?
3. **Failure mode** — what happens when this fails? Error handling consistency, recovery, observability (Sentry).
4. **Performance surface** — any hot paths? Hydration boundaries that ship too much client tree? Bundle weight contributions?
5. **a11y / motion / contrast** — does the surface honor `useReducedMotion` and the WCAG-AA opacity floor?

Pick the angles that match the theme:

- `state-management` → angles 1, 2, 4 + "Zustand subscription footprint and re-render reach".
- `error-handling` → angles 1, 2, 3 + "PII scrubbing surface coverage" (cross-check against `src/lib/sentry-pii.ts`).
- `i18n-locale-parity` → angles 1, 2 + "missing-key delta across all 14 locales" + "hardcoded strings in JSX/metadata/aria/alt".
- `animation-gating` → angles 1, 5 + "every `requestAnimationFrame` / canvas / framer-motion variant call site checked for `useReducedMotion`".
- `mock-vs-live` → angles 1, 2 + "mockApi vs orchestrator-URL drift; type contracts that pretend the orchestrator is shaped like the mock".
- `a11y-and-contrast` → angle 5 (deeply) + "every text utility checked against the `/60` opacity floor" + "form/keyboard navigation gaps".
- `react-19-purity` → angles 1, 2 + "synchronous setState in useEffect" + "impure Math.random/Date.now in render or useMemo".
- `performance` → angles 1, 4 + "client tree boundary placement (`'use client'` directives)" + "framer-motion reach and weight".

If `area` mode, every angle is bounded to files within the area's contexts.

### 3b. Sub-agent prompt template

Each sub-agent prompt should be **self-contained**. Use `Explore` (read-only) for all of them.

```
You are scanning the personas-web codebase for {angle name}.

Theme: {theme}
{If area mode:} Scope: only files under {area paths}
Background: {1 paragraph from CLAUDE.md / design.md relevant to the theme}

Specific questions:
1. {question 1 tailored to angle}
2. {question 2}
3. {question 3}

Report format (Markdown):
- Files inspected: {list, capped at top 30 by relevance}
- Observed shapes: {distinct patterns found, with file:line examples for each}
- Inconsistencies: {where shapes diverge — call out specific files}
- Outliers: {any single file doing it differently from the rest}
- Smell strength: 1-5 (1 = healthy, 5 = active drag on the codebase)
- Cross-references: {where this angle interacts with other parts of the system}

Budget: 30-60 minutes of equivalent work. Don't enumerate every match — sample
strategically and report shape, not exhaustive detail.
```

Run all sub-agents **in parallel**.

### 3c. Synthesize

Merge the sub-agent reports into a single pattern model. Look for:

- **Convergence** — multiple angles flagging the same module → high-confidence finding.
- **Conflict** — one angle calls something a strength, another calls it a weakness → investigate.
- **Surprise** — something none of the angles expected → likely the most valuable finding of the run.
- **Reach quantification** — every weakness has a concrete count: "47 files, 12 components, 6 app routes."

If sub-agent reports are thin (smell strengths all 1-2), the area is healthy in this theme. **Say so explicitly** and offer to either pick a different theme or downgrade the run to "passive scan." Don't manufacture findings.

### 3d. Output structure

After synthesis, you should have:
- 0–8 **weak-pattern findings** with reach, risk, effort, payoff.
- 0–4 **strong-pattern findings** worth codifying.
- 0–2 **tech-swap proposals** — only when smell strength is ≥4 AND swap unlocks payoff a refactor can't.
- 0–3 **structural-bug-class** findings — recurring bugs whose root is structural.

Cap total findings at **8**. If you have more, rank by `(reach × payoff) / (risk × effort)` and drop the bottom.

### 7c. Pre-flight checks

**Do NOT require a clean working tree.** Concurrent CLIs run on the same tree.

1. **Inspect the tree:**
   ```bash
   git status --short
   ```
2. **Classify each path** as in-flight by someone else / pre-existing in your touch zone / yours from this session. Surface conflicts to the user with a numbered menu (commit-first / commit-on-top default / abort).
3. **Capture validation baselines:**
   ```bash
   npm run typecheck         # baseline TS errors
   npm run lint              # baseline warning count
   npm run build             # only if a routing/config change is planned; expensive otherwise
   npm run test:e2e          # only if Playwright specs cover touched areas
   ```
   Record numbers in the ADR's `## Pre-flight baseline` section.

**Forbidden during pre-flight (and at every later phase):**
- `git stash` / `git reset --hard` / `git restore` / `git checkout --` on any path
- `git clean -f` / `git clean -fdx`
- `git add -A` / `git add .` / `git add -u`

If the working tree contains a path that conflicts with your edits and the user doesn't want to commit it first, abort the decision and queue it back to the backlog with a `blocked: working-tree-conflict` note.

### 7d. Atomic commits per rollout step

For each step in the ADR's Rollout section:

1. Apply the changes for that step.
2. Run the validation listed for that step.
3. **Compare to baseline** — TS errors must not increase, lint warnings must not exceed baseline + small rounding (5 max), tests must pass at the baseline rate.
4. If validation regresses → fix inline. Do NOT stack failing commits. Do NOT use `--no-verify` or `--amend`.
5. Stage **only the paths this step touched** — `git add path/one path/two`.
6. Commit using the personas-web format (CLAUDE.md §7):

   ```
   architect/<area>: <step title>

   Why: <root-cause or value>
   Risk: <low|med|high + rationale>
   Verified: <typecheck|lint|build|playwright>

   ADR: [[Architect/web/decisions/{date}-{slug}]]

   Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
   ```
7. Record the commit SHA in the ADR's Rollout section as you go.

### 7e. Final regression sweep

After the last step:
1. Run all validation commands fully:
   - `npm run typecheck`
   - `npm run lint`
   - `npm run build` (if config/routing touched)
   - `npm run test:e2e` (if surfaces with Playwright coverage touched)
2. Walk through the ADR's regression checklist.
3. **If any checklist item is unverified, do not mark the ADR as `shipped`.**

### 7f. Update ADR status

When all rollout steps are committed and regression checklist passes:
- Update ADR frontmatter: `status: shipped`, add `commits: [<sha>, ...]`.
- Move the entry in `Architect/web/backlog.md` from Pending to Shipped.

### 7g. Frontend changes — non-negotiable (per CLAUDE.md)

If any commit touches `src/**/*.tsx`:
- **i18n contract:** strings via `useTranslation()` + keys in `src/i18n/en.ts` + matching translations in all 13 non-en locales in the same commit.
- **Semantic Tailwind:** semantic tokens only; no raw white/black/hex; no `/40` or `/50` opacity on text.
- **Animation gating:** `useReducedMotion` import + short-circuit on every `requestAnimationFrame` / canvas / heavy framer-motion surface.
- **React 19 purity:** no sync setState in `useEffect`; no impure functions in render or `useMemo` factories.
- **Sentry PII:** route new `captureException`/breadcrumb data through `src/lib/sentry-pii.ts`.
- **Supabase guards:** every Supabase call must check URL + anon key.

### 7h. Visual verification

For UI-affecting decisions: launch `npm run dev`, exercise the affected route in a browser, confirm. State explicitly when you have NOT visually verified.

## Phase 7B: Codify strong patterns

Triggered for every strong pattern marked `codify` in Phase 6.

### 7B.a. Pick the vehicle

For each pattern marked `codify`, ask:

```
How should "{pattern title}" be codified? Pick one or more:

  1. lint-rule    — write a custom ESLint rule under eslint-rules/
  2. docs-design  — append a section to .claude/design.md
  3. docs-claude  — append a convention to .claude/CLAUDE.md (project rules)
  4. test-guard   — add a Playwright structural test that asserts the pattern
  5. multiple     — pick a combination (e.g. "1+3" = lint rule + CLAUDE.md docs)
```

**Vehicle rules of thumb:**
- Code-shape pattern (call-site discipline, hook usage) → `lint-rule` is strongest.
- Architectural boundary (module split, mock vs live, route shape) → `docs-claude`.
- Visual / design discipline (color, motion, spacing) → `docs-design`.
- Cross-file invariant detectable by file scan → `test-guard` (Playwright or a small Node script under `scripts/`).

### 7B.b. Lint rule vehicle

1. Read `eslint.config.mjs` and `eslint-rules/` to learn the project's custom-rule conventions (see `custom-a11y/no-low-text-opacity` and `custom-animation/require-animation-gating` for canonical examples).
2. Write a new rule under `eslint-rules/<rule-name>.{js,mjs}` matching existing rule shape.
3. Register the rule in `eslint.config.mjs`. Default severity: `warn`. Use `error` only if the user explicitly says "ship blocker."
4. Run `npm run lint` and capture the new warning count. If >500 new warnings, pause for guidance.
5. Commit: `architect/<area>: codify <pattern> as ESLint rule` with Why/Risk/Verified body.

### 7B.c. Docs vehicle (design or claude)

1. Read the target file (`.claude/design.md` or `.claude/CLAUDE.md`).
2. Find the right insertion point — for design.md: under tokens/components; for CLAUDE.md: under "Conventions (non-negotiable)".
3. Write the section: name, why it works, canonical example with `file:line` reference, anti-shape to avoid, optional pointer to the lint rule if `multiple`.
4. Keep it concise — 10-25 lines.
5. Commit: `architect/<area>: codify <pattern> in <file>`.

### 7B.d. Test guard vehicle

1. Read existing structural tests if any (`e2e/` Playwright specs, or any `scripts/` validators).
2. Write either:

## Notes on use

- **Cadence** — once a week is plenty. Architect runs are heavy; the backlog absorbs the inventory and resume mode amortizes the work.
- **Scan vs resume** — alternate. Scan to fill the queue, resume to drain it.
- **Coexist with uncommitted work.** Architect never requires a clean baseline. If a path it wants to edit already has uncommitted changes, surface the conflict to the user. Never `git stash`, never `git reset --hard`, never `git checkout --` someone else's work. Branching is opt-in.
- **Locale lockstep is real.** Any decision touching i18n keys ships with all 14 locales updated in the same commit (CLAUDE.md §1). The ADR's "i18n locale impact" section quantifies the translation work explicitly.
- **The branch question is real** — say `no` when you're already on a topic branch and the change belongs there, or when the change is small enough that a feature branch is overkill.
- **Conflict signal** — if a finding contradicts a `strong-pattern` already in the vault, treat it as the most interesting finding of the run. Either the entry is stale or the new finding is wrong.
- **Drift signal** — if 3 consecutive scans on different themes produce backlog items but zero get executed via resume, the user is using architect as a brainstorming tool, not a shipping tool. Surface this in self-reflection.
- **Tech swaps are the riskiest** — never propose a swap with reach ≥100 files unless smell strength is 5.
