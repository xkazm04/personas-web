# Knowledge Base
> Three lenses on the fleet's learned patterns + persistent memories — a dense sortable pattern table, an interactive SVG cluster graph, and a memory-card library with conflict review. **Route:** `/dashboard/knowledge` · **Nav label:** "Knowledge" · **Status:** Demo-only (mocks)

## What it does

The Knowledge surface answers *what has the agent fleet learned, and how confident is it?* It shows two kinds of knowledge:

- **Knowledge patterns** — observed cause/effect rules each persona has accumulated (e.g. "`web_search → summarize → email` succeeds 97% of the time at $0.023"). Every pattern carries a type (tool sequence / failure pattern / cost-quality / model performance / data flow), success & failure counts, a 0–1 confidence, average cost, average duration, and a last-seen time.
- **Memories** — persistent, accepted-or-pending operating rules the fleet keeps over time (throttle / schedule / alert / config / routing). Each memory has a 1–10 score, an active/pending/archived status, a usage count, and may be flagged as *in conflict* with another rule.

One page, three view modes selected by a tab switcher in the header (the choice persists across visits):

1. **Dense table** (default) — every pattern as a row, with sortable columns (type, key, agent, success, fails, rate, cost, duration, confidence, last seen), per-type filter pills, a stats strip (patterns / avg confidence / total success / total fails / avg cost), and a click-to-expand inline detail panel.
2. **Cluster graph** — patterns rendered as SVG nodes laid out in five radial clusters (one per knowledge type). Node size encodes confidence; nodes of the same persona are connected by dashed edges. Hovering a node highlights its persona's web and dims the rest; clicking opens a floating detail panel. Type filter pills + a stats strip (nodes / agents / clusters / avg confidence) sit on top.
3. **Memories** — a scrollable list of memory cards (icon, title, status pill, conflict badge, persona, score dots, uses, last-used). A filter bar narrows by type, and when any memory has a conflict a "resolve" button opens the **Batch Review modal** to accept/reject conflicting memories in bulk.

## How it works

**Data hook — `useKnowledgeData()`** (`useKnowledgeData.ts`). Returns `{ patterns, memories, loading, error }`. Mode is decided once per session from `authStore.isDemo`:
- **Demo mode** (this repo's default): `useState` initializers seed `MOCK_KNOWLEDGE_PATTERNS` / `MOCK_MEMORIES`, `loading` starts `false`, and the effect early-returns (`if (useMock) return;`, `useKnowledgeData.ts:128`). No fetch runs.
- **Real/Supabase mode**: the effect `Promise.all`s `getSyncedKnowledgePatterns()`, `getSyncedMemories()`, and `api.listPersonas()`, then maps the synced rows into the web display shapes (`mapSyncedPattern` / `mapSyncedMemory`), resolving `personaId → name` via the personas list. Unknown `knowledge_type` / memory `category` tokens are coerced to a safe default (`tool_sequence` / `config`) by `normalizeKnowledgeType` / `normalizeMemoryType`. Synced memories are always `status: "active"`, `usageCount: 0`, `hasConflict: false` (the desktop sync projection carries no accept/archive/conflict state). Errors are captured to Sentry with `tags: { scope: "useKnowledgeData" }` and surfaced via `error`.

**Page composition — `page.tsx`.** `KnowledgeGraphPage` holds `activeVariant` (`"dense-table" | "cluster-graph" | "memories"`). On mount it restores the persisted variant from `localStorage[KNOWLEDGE_VIEW_KEY]` inside `queueMicrotask(...)` to keep the `setState` out of the effect body (React 19 rule, `page.tsx:41`); `selectVariant` writes it back. The switcher is a real ARIA `role="tablist"` with roving tabindex — Arrow keys wrap, Home/End jump (`handleTabKeyDown`). While `loading`, three `SkeletonCard`s render; `error` renders a `DashboardErrorBanner`. The active panel mounts exactly one of the three view components, passing `patterns` or `memories`.

**View 1 — Dense table (`KnowledgeDenseTable.tsx` + `knowledge-dense-table/`).** Local state: `sortField` (default `confidence`), `sortDir` (default `desc`), a `Set<KnowledgeType>` of active type filters, and `selectedPattern`. `sortedPatterns` filters by the type set (empty set = all), then sorts via the in-file `compareKnowledgePatterns` comparator (`successRate` uses `knowledgeSuccessRate`, `lastSeen` parses dates per-compare). `stats` is memoized off the unfiltered list. Columns are built by `buildKnowledgeColumns(t.knowledgePage)`. Rendering: `KnowledgeDenseTopBar` (stats + filter pills) → `KnowledgePatternTable` (sticky `KnowledgeSortHeader` row + `KnowledgePatternRow`s inside `<AnimatePresence mode="popLayout">` with `layout`) → an `<AnimatePresence>` `KnowledgePatternDetailPanel` that expands height-auto below the table. Each `KnowledgePatternRow` is a `<motion.button>` with zebra striping, a confidence mini-bar colored by `knowledgeConfidenceColor` (≥0.7 emerald, ≥0.4 amber, else rose), and formatted cost/duration/relative-time cells.

**View 2 — Cluster graph (`KnowledgeClusterGraph.tsx` + `knowledge-cluster-graph/`).** Local state: `dimensions` (from a `ResizeObserver` on the container), `selectedPattern`, `hoveredPattern`, `activeFilter` (`"all"` or one type). `filteredPatterns` narrows by the active filter; `nodePositions` and `edges` are recomputed from it; `highlightedIds` is the set of nodes sharing the hovered node's persona; `stats` is over the *unfiltered* list. Layout math lives in `knowledgeClusterLayout.ts`:
- `computeKnowledgeNodePositions` groups patterns by type, places each cluster center on a circle of radius `min(w,h)*0.34` at the type's `clusterAngle` (0/72/144/216/288°), and arranges members around the cluster at a per-node distance scaled by confidence (`spreadRadius * (0.4 + confidence*0.6)`).
- `computeKnowledgeEdges` connects every pair of patterns owned by the same persona (a per-persona complete graph), tagged with `persona` for edge color.

`KnowledgeClusterSvg` draws `<motion.line>` edges (animated `pathLength`, dashed unless their endpoint is hovered) then `KnowledgeGraphNode`s. Each node is a `<motion.g>` with a circle whose diameter scales `28→58px` with confidence, a centered lucide icon in a `<foreignObject>`, and a truncated label with a `var(--background)` stroke halo for legibility. `KnowledgeClusterLabels` floats the five type names at the cluster angles (only when `activeFilter === "all"`); `KnowledgeClusterLegends` shows the persona color key (bottom-left) and a node-size legend (top-left); `KnowledgeClusterTopBar` carries stats + the All/per-type filter pills (selecting a filter also clears the selection). The selected node's panel is `KnowledgeClusterDetailPanel`, a floating glass card top-right with success/fail/rate, cost/duration/confidence, an animated confidence bar, the description, and a relative last-seen.

**View 3 — Memories (`MemoriesView.tsx` + `memories-view/`).** Local state: `filter` (`"all"` or a memory type), `resolvedIds` (a `Set<string>`), and `modalOpen`. `visibleMemories` overlays `hasConflict: false` onto any memory whose id is in `resolvedIds`; `filtered` narrows by type; `activeConflicts` is the still-conflicting subset. `MemoriesToolbar` renders the shared `FilterBar` (per-type counts) plus a rose "resolve conflicts" button (only when `activeConflicts.length > 0`). Each `MemoryCard` shows the type icon, title, status pill (`statusConfig`), a conflict badge when flagged, the persona chip, `ScoreDots` (5 dots, `filled = round(score/10 * 5)`, min 1), a uses count, and a relative last-used. The list container uses `content-visibility: auto` per card for long-list perf. Clicking resolve opens `BatchReviewModal`.

**Batch Review modal (`src/components/dashboard/BatchReviewModal.tsx` + `batch-review-modal/`).** Lives outside the knowledge folder but is *only* consumed by `MemoriesView`. A focus-trapped (`useFocusTrap`) `role="dialog"` listing each conflicting memory as a `ConflictDecisionCard` with accept/reject toggles. It tracks per-id `decisions`; the Apply button is disabled until at least one decision exists and reports `pending / total`. Closing with un-applied decisions triggers a `BatchReviewDiscardDialog` confirm. Escape cancels the discard dialog first, otherwise requests close. `onApply` lifts the decided ids back to `MemoriesView.handleApply`, which adds them to `resolvedIds` (clearing their conflict locally — no persistence). The open/reset transition uses the prev-state pattern (`if (open !== prevOpen) { … }`) rather than a `setState`-in-effect.

## Data & state
- **Source:** `MOCK_KNOWLEDGE_PATTERNS` (8 patterns) and `MOCK_MEMORIES` (60 seeded memories, ~12% flagged in conflict) in `src/lib/mock-dashboard-data.ts`. Both use timestamps relative to `Date.now()` at module load so relative times stay fresh; `MOCK_MEMORIES` is built with a seeded RNG (`seededRandom(555)`) for deterministic content. In real mode the data comes from `getSyncedKnowledgePatterns()` / `getSyncedMemories()` (`src/lib/supabaseApi.ts:673` / `:646`) mapped through `mapSyncedPattern` / `mapSyncedMemory`.
- **Stores:** No Zustand store for knowledge data — all data state is local to `useKnowledgeData`'s `useState`. `authStore.isDemo` selects mock vs. live mode. View-mode selection persists in `localStorage[KNOWLEDGE_VIEW_KEY]` (`"personas-dashboard-knowledge-view"`, `src/lib/constants.ts:34`), not a store.
- **API routes:** None. Data is either in-process mocks (demo) or direct Supabase queries (live) — there are no `src/app/api/*` routes behind this surface.
- **Types:** `KnowledgePattern`, `MemoryItem`, `MemoryAction`, `MemoryStatus` from `src/lib/mock-dashboard-data.ts`; `SyncedKnowledgePattern` / `SyncedMemory` from `src/lib/supabaseApi.ts`; `KnowledgeData` (hook return) from `useKnowledgeData.ts`; per-view `KnowledgeType` / `SortField` / `ColumnDef` / `NodePosition` / `FilterKey` / `BatchDecision` in the subfolders.

## Integration points
- **Dashboard shell & nav.** Registered in `navItemDefs` (`src/components/dashboard/DashboardNavigation.tsx:35`) with `key: "knowledge"`, `labelKey: "knowledge"`, `Brain` icon, `href: "/dashboard/knowledge"`. Renders inside the shared dashboard layout (sidebar / mobile bottom nav). Nav label resolves from `t.dashboard.knowledge`.
- **Shared dashboard primitives.** Uses `SkeletonCard`, `DashboardErrorBanner`, `GradientText` (page chrome); `FilterBar` (memories type filter); and `BatchReviewModal` (+ its `batch-review-modal/` children and `useFocusTrap`). The table and graph use bespoke top bars / pills rather than the shared `FilterBar`, because their pills carry per-type icons and toggle a `Set`, not a single active key.
- **Formatting utilities.** Two layers, intentionally noted: the cluster detail panel imports the shared `formatCost`/`formatDuration`/`relativeTime` from `src/lib/format.ts`, while the dense table uses its own near-duplicate `knowledgeDenseFormat.ts`. (`derived.ts` holds a *third* copy that nothing imports — see gotchas.)
- **i18n.** All page copy under `t.knowledgePage.*` (view labels, column/stat labels, legends, `types.{tool_sequence,failure_pattern,cost_quality,model_performance,data_flow}`); memory copy under `t.memoriesPage.*` (filters, status, `uses`/`totalCount`/`empty` with `{n}`, and the whole `conflicts.*` modal vocabulary); the conflict badge uses `t.dashboardUi.conflict`; close buttons use `t.common.close`. Interface at `src/i18n/en.ts:433` (knowledge) and `:397` (memories).
- **Persona colors.** `PERSONA_COLORS` (graph edges/legend) is a hardcoded map of the five mock persona names → hex. Live personas not in this map fall back to `#64748b` for edges and won't appear in the legend.

## Conventions & gotchas
- **No reduced-motion gating anywhere in this feature — this is a real gap.** Every view animates heavily (SVG edge `pathLength` draw-on, node `scale: 0 → 1`, confidence bar width, row `layout`/`AnimatePresence`, panel transitions) yet **no file in `src/app/dashboard/knowledge/` imports `useReducedMotion`**. The repo's `custom-animation/require-animation-gating` rule keys on `requestAnimationFrame`/`cancelAnimationFrame` (which framer-motion abstracts away), so it doesn't flag these — but `CLAUDE.md` still requires GPU-intensive motion to be gated. The cluster graph (dozens of `motion.line`/`motion.g`) is the most expensive offender. If you touch these components, add `useReducedMotion()` and short-circuit durations/`pathLength`.
- **`derived.ts` is dead code.** Despite its docstring claiming it caches per-row fields "so neither the table nor the graph has to recompute…", nothing imports `deriveKnowledgePatterns`, `DERIVED_KNOWLEDGE_PATTERNS`, or `relativeFromNow`. The live table sorts with its own inline `compareKnowledgePatterns` (re-parsing dates per compare) and formats with `knowledgeDenseFormat.ts`. Don't trust the comment; either wire `derived.ts` in or delete it.
- **Three duplicate formatter/config sets.** `formatCost`/`formatDuration` exist in `@/lib/format`, in `knowledgeDenseFormat.ts` (as `formatKnowledge*`), and in `derived.ts`. `KNOWLEDGE_TYPE_CONFIG` (dense) and `KNOWLEDGE_CLUSTER_TYPE_CONFIG` (graph) are near-identical type→icon/color maps that differ only by the graph's extra `clusterAngle`. Keep them in sync when changing a type's icon/color, or consolidate.
- **Unused i18n keys.** `t.knowledgePage.tokens`, `t.knowledgePage.retries`, and `t.memoriesPage.seeAll` are declared in the `Translations` interface but referenced nowhere in this feature. Don't add columns for them assuming they're already wired.
- **Stats are computed over the *unfiltered* list, by design.** Both top bars show totals/averages for all patterns even when filters narrow the visible rows/nodes. This is intentional (filters affect the view, not the headline numbers) — don't "fix" it to match the filtered set without checking intent.
- **i18n 14-locale lockstep.** Every user-facing string is a `t.*` key (including `aria-label`s via `t.common.close` and `viewSwitcherLabel`). Adding a label means adding it to `en.ts` first and hand-translating into all 13 other locales in the same commit. Interpolation uses literal `{n}` placeholders replaced via `String.replace` at the call site (memories counts).
- **Semantic Tailwind tokens.** Use `border-glass`, `bg-white/[0.0x]`, `text-foreground`, `text-muted-dark`. Data-driven colors are the intentional exception: `config.color`/`PERSONA_COLORS`/`knowledgeConfidenceColor` are applied via inline `style` (hex), not tokens, because they encode pattern type / persona / confidence band.
- **React 19 purity.** The view-restore `setState` is deferred via `queueMicrotask` out of the effect body (`page.tsx:45`); `BatchReviewModal`'s open/reset uses the prev-state pattern. But `relativeKnowledgeTime` and `relativeTime` call `Date.now()` directly in render — acceptable here only because they feed display text, not derived/memoized state. Don't move clock reads into a `useMemo` factory.
- **Conflict resolution is local-only.** `MemoriesView.handleApply` just adds ids to a `resolvedIds` set in component state; nothing persists, and synced memories never carry `hasConflict: true` anyway, so the resolve flow is effectively demo-only.
- **Demo-only caveat.** In this repo `isDemo` is effectively always true, so the live `getSynced*` branch and the `loading`/`error` paths render only against real Supabase data — exercise them with mock toggles, not by editing the page.
